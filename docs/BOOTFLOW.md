# PhoenixOS Boot Flow

This document traces every step from power-on to an interactive shell or desktop session.

---

## Stage 1 — Firmware (BIOS/UEFI)

1. CPU executes the reset vector at `0xFFFFFFF0` (real mode)
2. BIOS POST: memory test, device enumeration
3. BIOS reads the MBR from the first bootable disk and executes it
4. **(UEFI)**: Firmware locates the EFI system partition, loads the GRUB EFI binary

---

## Stage 2 — GRUB2 Bootloader

GRUB2 is the second-stage bootloader stored in `/boot/grub/`. PhoenixOS uses
the **Multiboot2** specification.

1. GRUB reads `boot/grub/grub.cfg` (see `boot/grub.cfg`)
2. Presents the boot menu entry: **"PhoenixOS — EmberKernel"**
3. Loads `kernel/ember.elf` into memory and parses its ELF sections
4. Constructs a Multiboot2 information structure containing:
   - Physical memory map (E820)
   - Framebuffer address, dimensions, pitch, bits-per-pixel
   - Command-line arguments
5. Switches to **32-bit protected mode** (flat segments, paging disabled)
6. Calls `_start` in `kernel/arch/x86_64/boot/boot.asm` with:
   - `eax = 0x36d76289` (Multiboot2 magic)
   - `ebx = physical address of Multiboot2 info structure`

## Stage 3 — `boot.asm` (32-bit → 64-bit transition)

Source: `kernel/arch/x86_64/boot/boot.asm`

### 3.1 Verify Multiboot2 magic

```asm
cmp eax, 0x36d76289
jne .no_multiboot
```

If the magic number is wrong, the CPU halts with an `'M'` error indicator in
VGA text memory.

### 3.2 Check CPU capabilities (CPUID)

- Queries `CPUID 0x80000001` for the `LM` (Long Mode) bit in EDX
- Halts with `'L'` error if the CPU is 32-bit only

### 3.3 Set up initial page tables

Four 4 KB page-aligned tables are reserved in BSS (`pml4_table`, `pdpt_table`,
`pd_table`):

```
PML4[0] → PDPT[0] → PD[0] → 2 MB huge page (phys 0x000000, present+RW+PS)
```

This identity maps the first 2 MB of physical memory so that `boot.asm` code
keeps running after paging is enabled.

### 3.4 Enable PAE and Long Mode

```asm
; PAE
or cr4, (1 << 5)
; Long Mode Enable (LME) via EFER MSR
mov ecx, 0xC0000080
rdmsr
or  eax, (1 << 8)
wrmsr
; Enable Paging
or cr0, (1 << 31)
```

### 3.5 Load 64-bit GDT and far-jump

A minimal 64-bit GDT (null + kernel code + kernel data + user code + user data)
is defined in `boot.asm`. After `lgdt`, a far jump reloads CS with the 64-bit
code segment selector (0x08) and the CPU enters **IA-32e (64-bit) mode**.

### 3.6 64-bit stub

```asm
bits 64
.long_mode:
    mov ax, 0x10    ; kernel data segment
    mov ds/es/fs/gs/ss, ax
    mov rsp, boot_stack_top
    mov edi, edi    ; zero-extend saved multiboot ptr
    call kernel_start
```

The boot stack is 16 KB (`boot_stack_bottom` .. `boot_stack_top` in BSS).

## Stage 4 — `kernel_start()` (C entry point)

Source: `kernel/main.c`

```c
void kernel_start(u32 multiboot_info_phys) {
```

Called from `boot.asm` with the Multiboot2 info structure physical address in `edi`/`rdi`.

### 4.1 Early serial console

`serial_init()` — COM1 at 0x3F8, 115200 baud, 8N1.
`early_printk("EmberKernel " VERSION " booting...")` — visible immediately in QEMU's `-serial stdio`.

### 4.2 GDT reload

`gdt_init()` rebuilds the GDT in kernel BSS (replacing the minimal boot GDT)
and sets up the TSS for ring-0 stack switching. `load_tss()` loads the TSS
selector with `ltr`.

### 4.3 IDT setup

`idt_init()` installs 256 interrupt gates:
- 0–31: CPU exception handlers (generated by macros in `startup.asm`)
- 32–47: Hardware IRQ handlers (after PIC remapping)
- 128 (0x80): Legacy int80 syscall handler

### 4.4 Physical Memory Manager

`pmm_init(mmap, count)` — parses the Multiboot2 memory map, marks all
available pages as free in the bitmap, then marks the kernel's physical pages
as used.

### 4.5 Virtual Memory Manager

`vmm_init()` — rebuilds the kernel page tables with the final higher-half
mappings (kernel at `0xFFFFFFFF80100000`), maps the framebuffer, and switches
`cr3` to the new PML4.

### 4.6 Framebuffer console

`fb_init(addr, width, height, pitch, bpp)` — from Multiboot2 framebuffer tag.
`console_init()` — sets up cursor, clears screen, ready for `printk`.

### 4.7 Interrupts and timer

`pic_init()` — remaps 8259A: IRQ 0–7 → vectors 32–39, IRQ 8–15 → 40–47.
`pit_init(100)` — channel 0 at 100 Hz, connected to IRQ 0.
`sti()` — enable interrupts.

### 4.8 Scheduler

`sched_init()` — creates the idle task (PID 0) running `idle_loop()`.

### 4.9 Syscall gate

`syscall_init()` — programs `LSTAR` (kernel entry point), `STAR` (segment
selectors), and `SFMASK` (flags to clear on entry).

### 4.10 Root filesystem

`vfs_mount("/", ext2_driver, root_device)` — mounts the ext2 root partition
passed via the kernel command line or detected from the disk.

### 4.11 Launch Kindle (PID 1)

`task_create("kindle", kindle_entry, NULL, 16)` creates the init task.

### 4.12 Idle loop

`kernel_start` falls into the scheduler idle loop:

```c
while (1) { sti(); hlt(); }
```

## Stage 5 — Kindle (Init, PID 1)

Source: `userspace/init/kindle.c`

Kindle is PhoenixOS's PID 1 init system. It displays an OpenRC-style startup
animation on the console so the user can see each service being started and
whether it succeeded or failed.

### 5.1 Boot splash

When Kindle starts it prints a coloured ASCII-art PhoenixOS banner to stdout:

```
  ██████╗ ██╗  ██╗ ██████╗ ███████╗███╗  ██╗██╗██╗  ██╗ ██████╗ ███████╗
  ...
  PhoenixOS — Kindle init v1.0   (custom UNIX-like OS)
```

### 5.2 Filesystem mounts

Before reading any service files Kindle mounts the three essential virtual
filesystems and prints a status line for each:

```
 [ * ] Starting proc filesystem ...                   [ ok ]
 [ * ] Starting sysfs filesystem ...                  [ ok ]
 [ * ] Starting devtmpfs ...                          [ ok ]
```

### 5.3 Service startup animation

1. Reads service definitions from `/etc/kindle/services/*.svc`
2. Performs dependency-ordered startup
3. For each service being started a line is printed before forking:

   ```
    [ * ] Starting phx-svcmon ...                     [ ok ]
    [ * ] Starting tty-manager ...                    [FAIL]
   ```

   - `[ ok ]` (green) — process forked and did not exit within 20 ms
   - `[FAIL]` (red)  — fork failed, or process exited immediately (exec error)
   - `[ !! ]` (yellow) — warning / skipped (used when reporting dependency waits)

4. A summary info line is printed when all initial services have been launched:

   ```
     ~~   Loaded 4 service(s) from /etc/kindle/services
     ~~   Kindle init complete — system is up
   ```

### 5.4 Supervision loop

After the initial startup pass Kindle enters a supervision loop (`supervise()`),
restarting services as configured by their `restart=` policy, and printing
`[FAIL]` notices to the console whenever a supervised service exits unexpectedly.

### 5.5 Session detection

- Console: launches `login` on `/dev/tty0`, which runs PyreShell
- Desktop: launches the session manager

---

## Stage 6 — Desktop Session (AshDE)

1. **Session Manager** (`desktop/sessionmgr/sessionmgr.c`)
   - Sets up `PATH`, `DISPLAY`, `DESKTOP` environment variables
   - Forks and execs the WM and Panel

2. **AshDE Window Manager** (`desktop/wm/main.c`)
   - Opens `/dev/fb0`, maps framebuffer
   - Draws teal desktop background
   - Starts polling for mouse/keyboard events

3. **Panel** (`desktop/panel/panel.c`)
   - Draws 28-pixel taskbar at the bottom
   - Starts clock update loop

4. **Login greeter** (optional — can run before session manager)
   - Displays ASCII art Phoenix logo
   - Accepts username / password
   - Calls `setuid` / `setgid`, then `exec`s session manager

## Boot Time Summary

| Stage | Approximate wall-clock time |
|-------|----------------------------|
| BIOS POST | 1–3 seconds |
| GRUB menu | 0–5 seconds (configurable timeout) |
| `boot.asm` (long mode setup) | < 1 ms |
| `kernel_start` (Phase 1–7) | < 50 ms |
| Kindle service startup | 0.5–2 seconds |
| Desktop session ready | 1–3 seconds |
| **Total** | **~4–10 seconds** (QEMU) |
